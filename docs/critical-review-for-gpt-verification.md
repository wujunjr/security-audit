## 0. 执行摘要（给复核者的 10 条高压质疑）

1. **商业假设尚未被任何真实订单或付费意向数据支撑**；当前仓库只有 landing + 样例报告 + mailto 表单，却已规划自动扫描、监控订阅、专家复核多层产品——存在「用工程复杂度代替市场证据」的风险。
2. **「AI-built app 开发者愿为上线前配置体检付费」** 与 **「愿为人工复核付费」** 是两个不同假设；规格把后者列为收入优先级 #1，但成功指标同时要求扫描量与订单，**未规定若只有扫描无订单时的止损动作的强制执行机制**（虽有文字，缺运营定义）。
3. **品类极度拥挤**：Mozilla Observatory、securityheaders.com、Internet.nl、各类免费 header 检查与 AI 聊天即可解释修复步骤。ShipAudit 的差异化若仅靠「中文解释 + AI 摘要」，**护城河接近零**。
4. **$99 founding review 与后续 ¥49 / ¥19/月 / ¥299 价位混用** 尚未在规格中统一货币、客群与交付边界，存在定价叙事混乱与「廉价自动报告稀释高价专家服务」的风险。
5. **被动配置体检的结果对「上线是否安全」解释力有限**；若营销话术稍有越界，将同时伤害信任与法律风险。规格有免责声明，但**样例报告与首页是否严格执行**需对抗性检查。
6. **Cloudflare Workers 免费层 CPU/subrequest 约束** 与「10 秒出报告、多跳重定向、HTML 截断解析、DNS 校验」并存；规格已警告，但**尚未用真实目标集证明可在 Free 层稳定完成**。
7. **SSRF 规范写得很强**，属于正确优先级；审查关注的是：**实现遗漏单点即全盘沦为攻击跳板**，且「附加 DNS 邮件检查」会增加解析与错误面。
8. **Chrome 扩展路线** 对当前阶段适配度低；若有人再次推动「扩展优先」，将直接违反已验证的商业顺序——本文明确反对现阶段投入。
9. **「Verified report 可分享」与「徽章/品牌背书」** 若缺少服务端独立取证与防伪，会沦为可伪造营销贴纸，反噬安全产品信誉。
10. **目标用户排序**（英文 indie 优先，中国出海第二）与运营者约束（大陆、公众号、收款弱、中文内容资产）之间存在**获客渠道错配**；规格未充分回答「第一批付费用户从哪来」。

请 GPT 复核时：不要只说「同意总体方向」；请对 **§1–§8 的编号问题** 给出可执行的接受/拒绝/改规格建议。

### 2.2 差异化是否站得住

规格声称相对本地工具的价值在于：免安装、统一优先级、AI 栈解释、可分享记录、人工复核。

**对抗性质疑：**

1. **免安装**：Observatory / securityheaders 已满足。
2. **统一优先级**：可用固定规则表实现，竞品也能抄；优先级若与真实风险脱节（例如缺 Permissions-Policy 与缺 HTTPS 权重不当），专家会被嘲笑。
3. **AI 栈解释（Supabase/Firebase/Vercel）**：高价值，但规格 MVP 规则表仍以经典 header/TLS 为主；**「AI-built」叙事与「经典 header 扫描」内容错配**。若首页强调 AI-built，报告却几乎全是 CSP/HSTS，用户会感到 bait-and-switch。
4. **可分享记录**：有价值，但需防伪与范围声明，否则变成「自助贴金」。
5. **人工复核**：真差异化，却最难规模化；规格未写清 **SLA、交付物模板、每人时上限、退款条件、利益冲突（是否帮改代码）**。

**请 GPT 回答：** MVP 自动扫描是否必须包含至少 N 条「AI/BaaS 托管特有」检查（例如公开环境变量模式、supabase URL 暴露形态等，且仍保持被动）？若否，是否应弱化首页 AI-built 话术？

### 2.3 目标用户与渠道错配

**批评：** 操作者在中国大陆，收款与品牌资产偏中文；英文冷启动通常依赖 PH/HN/Twitter/SEO。规格写了成功指标，**没写获客 weekly 动作与预算**（月预算若仅 ¥100–500，PH 以外几乎只能内容/SEO）。

## 3. 产品范围与「诚实边界」批判

### 3.1 品类诚实性（支持，但执行易滑落）

审查**支持**以下冻结：

- 被动配置体检，非漏洞扫描 / 渗透 / 等保认证  
- 禁止端口扫描、利用、登录态、API 深审等  

**仍批评：** 产品名 ShipAudit、副标题 Pre-launch security check、分数 A–F，用户仍会读成「安全认证」。仅靠 methodology 页不够。

**要求：**

1. 报告首页固定 **Scope / Non-goals** 模块（机器可读 + 人读）。  
2. 禁止 UI 文案：safe / 通过等保 / 无漏洞 / certified。  
3. 分数旁永久副文案：配置卫生评分，非风险穷尽。

**请 GPT 审核：** 当前样例报告与首页文案是否已满足；若未读页面，请列出必须出现的英文禁用词与必现声明句。

### 3.2 检查项集合的内部张力

对话列出的 MVP 检查包括：HTTPS 跳转、安全头、Cookie、CORS、Referrer/Permissions-Policy、security.txt/robots/sitemap、SPF/DMARC、重定向链、Server 头等。

规格将 security.txt、SPF、DMARC 等作**附加不计分**——这点合理。

**批评：**

1. **SPF/DMARC 与「网站上线」弱相关**；对纯应用站可能造成「邮件分低但站点可上线」的认知噪音。是否应默认折叠，仅在检测到 MX 时展示？  
2. **CORS** 在「仅扫首页 HTML 文档响应」时经常无意义；规格把部分 API CORS 放到验证后——良好。对话若让人以为 MVP 全面 CORS，会误导实现者。  
3. **robots/sitemap** 安全价值低，偏「上线卫生」；放在安全产品中可能显得拼凑。  
4. **CSP 评分权重高（规格 18 分）** 对营销落地页不公：无 CSP 的静态页未必不能上线；可能制造误报式低分，增加人工复核的「解释成本」。

**请 GPT 回答：** 是否调整评分权重或引入「站点类型 profile」（marketing / app）？若增加 profile，是否违反「一次只做一个阶段」？

### 3.3 与「AI-built」叙事的缺口（再次强调）

若核心付费点是 AI 快速上线者，审查期望报告中出现例如（仍保持被动、有限）：

- 疑似客户端暴露的密钥/token 模式（首页 HTML/JS 截断内）  
- source map 引用  
- 明显第三方分析/聊天脚本列表  
- 托管平台指纹的**修复建议链接**（Vercel/Netlify 文档）  

规格对敏感文件探测限制在验证后——正确。但**免费层若完全没有 AI/BaaS 特色 finding**，叙事破产。

**请 GPT：** 列出「免费层至少 3 条 AI-built 特色、且安全可做的被动检查」或明确删除 AI-built 主叙事。

## 4. 安全工程批判（SSRF 与滥用）

### 4.1 规格质量

第 6 节 SSRF/输入/重定向规范整体**达到可实施的高标准**。审查肯定：manual redirect、每跳 DNS、禁止 IP、私网与 metadata、响应上限、无用户 Cookie、CORS 禁止 `*`。

### 4.2 仍可能被实现者漏掉的攻击面

请 GPT 确认是否应写入规格「显式测试用例」：

| 攻击/滥用 | 说明 |
|-----------|------|
| DNS Rebinding | 校验时公网、fetch 时变私网；是否要求连接后二次校验或短 TTL 策略？ |
| 超大 header / 超多 Set-Cookie | DoS Worker CPU |
| 国际化域名与同形异义 | 钓鱼展示 vs 实际请求主机 |
| 开放重定向链消耗 | 5 跳内指向大文件或慢源 |
| 把扫描器当代理 | 对第三方 URL 的放大（与限流、缓存策略耦合） |
| 报告 token 枚举 | 未认证可读报告导致信息泄露 |
| 自扫描与缓存投毒 | 规格禁止扫自身；缓存键是否含规范化 host？ |

**请 GPT：** 指出上表哪些必须进 phase 1 验收测试，哪些可延期。

### 4.3 「安全产品」自身的信任

- 固定 UA 含联系邮箱：好，但是否准备好处理 abuse 投诉？  
- 不存完整 HTML：好；则**证据链**如何在争议时证明「当时头是什么」？是否存规范化 header 快照哈希？  
- 专家复核若需登录态：规格禁止用户提交 Cookie——正确；则人工如何做？是否仅限公开面 + 用户录屏？**交付边界必须写清，否则 $99 不可售。**

## 5. 技术架构批判

### 5.1 Web + Worker + D1 + KV + Turnstile

对 **Verified 公网体检**，该架构合理。审查**同意**作为主路径。

**批评：**

1. Free 层 10ms CPU：CSP 解析 + 多规则 + JSON 是否超标，**零实测**。  
2. 规格允许升级 Workers Paid ~$5——与「月预算极低」的操作者画像需对齐。  
3. 「GLM 仅归纳已有 finding」——防幻觉依赖提示词与校验；需 **schema 校验与禁止新增 finding ID** 的硬编码闸门，否则合规声明被模型破坏。  
4. monorepo 已存在 contracts——良好；但 **rules 包尚未见**，避免规则写死在路由里导致不可测。

### 5.2 Chrome 扩展 + Cloudflare「同一项目」

| 命题 | 裁决 |

**请 GPT：** 用规格补丁形式起草「§X Chrome Extension（非 MVP）」五条以内硬约束（含权限与报告标签）。

审查同意：KV 不作订单唯一事实源；长期 license key 不适合扩展；webhook 需验签幂等退款。
